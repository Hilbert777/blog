### 负载均衡

负载均衡（Load Balancing）是计算机网络中一种将工作负载（如网络流量、请求、计算任务）分散到多个服务器或网络资源上的技术。它的核心目标是提高系统的整体处理能力、可用性、可扩展性，并避免单点故障。

#### 为什么需要负载均衡

在大型网站或服务中，如果只有一台服务器处理所有请求，当访问量激增时，服务器可能过载，响应变慢甚至崩溃。一旦这台服务器故障，整个服务就会中断。

#### 负载均衡的工作原理
负载均衡器通过某种算法，将客户端请求转发到选定的后端服务器，并将服务器的响应返回给客户端。它通常工作在两种模式：

##### 四层负载均衡（基于传输层，如 TCP/UDP）
根据 IP 地址和端口号进行转发，不解析报文内容。效率高，但无法基于 HTTP 头、URL 等做精细化分发。
典型实现：LVS（Linux Virtual Server）、F5 硬件设备。

##### 七层负载均衡（基于应用层，如 HTTP/HTTPS）
可以解析 HTTP 请求，根据 URL、Cookie、Host 头等做出智能路由。支持内容缓存、SSL 卸载等高级功能。
典型实现：Nginx、HAProxy、Apache 反向代理。

#### 负载均衡实例分析

第 6 跳的这种情况通常由等价多路径（ECMP，Equal-Cost Multi-Path）路由导致。
在网络核心层，为了增加带宽和冗余，路由器之间会配置多条“等价”路径。当数据包经过时，路由器会根据某种哈希算法将流量分散到多条链路上，从而实现流量分担。

第 6 跳代表一个逻辑上的“下一跳”，但实际存在多个物理设备或接口。三个探测包被分别送往不同的路径，因此收到了不同设备（或同一设备的不同接口）的回应。

### 隐形墙（丢弃ICMP包）

第 8 到 15 跳全部超时，但第 16 跳成功了。其原因是腾讯公司在核心路由器上配置了丢弃 ICMP 包的策略。

那么为什么要采取这种策略呢，主要是出于安全考虑

#### ICMP 包与 traceroute 的原理
traceroute 默认发送 UDP 包（或 ICMP Echo 请求，取决于实现），并设置递增的 TTL（生存时间）。每经过一跳，TTL 减 1，当 TTL 变为 0 时，该跳的路由器会向源发送一个 ICMP 超时消息。traceroute 正是通过接收这些 ICMP 超时消息来记录每一跳的 IP 地址。

如果路由器不发送 ICMP 超时消息（即丢弃了触发的包且不回应），traceroute 就会显示 *。

#### 为什么网络设备会丢弃 ICMP 包
##### 安全与防探测
**隐藏网络拓扑**：攻击者常利用 traceroute 等工具绘制网络结构。丢弃 ICMP 回应可以防止内部网络结构被轻易发现。

**减少攻击面**：ICMP 协议历史上被用于多种攻击（如 ICMP 洪水、Smurf 攻击、隧道攻击）。限制 ICMP 响应能降低风险。

##### 性能考虑
骨干网路由器的主要任务是高速转发数据包，生成 ICMP 超时消息需要 CPU 介入，会占用处理资源。在高流量场景下，路由器可能选择不回应，以保证转发性能。

一些运营商路由器会限速或直接禁止 ICMP 超时消息的发送。

##### 避免放大攻击
如果路由器对每个 TTL 超时的包都回复 ICMP，攻击者可以利用伪造源 IP 的方式，让路由器向受害者发送大量 ICMP 包，形成反射放大攻击。丢弃这类回应可以防止路由器成为攻击的“放大器”。

### 失败原因分析
#### 国际链路严重拥塞
第 16 跳出现的 358ms 延迟是典型的队列溢出前兆。当路由器缓冲区填满时，它会丢弃新的数据包，并且处理现有数据包的速度极慢（导致高延迟）。中美之间的海底光缆带宽有限，且在特定时间段（如工作时间）极易拥塞。这种情况的可能性比较大。
#### 防火墙拦截 
这是一个下意识反映出来的原因，但是仔细分析可能性并不大，一方面GFW长期以来没有拦截 Comcast 的先例。即使是拦截了，一般而言会直接阻断会导致更早的超时，而不是在第 16 跳才消失。

### 为什么浏览器可以访问但是trace失效？
一言以蔽之：两者采用的协议不同！这里我们做一个比喻。浏览器访问使用的是TCP协议，就像是持有正规签证的旅客，走海关通道，虽然检查严格，但只要手续齐全就能通过。Traceroute使用的是ICMP/UDP协议，像是一个在边境线上的可疑人士，一眼看上去行为就很可疑（探测网络拓扑），直接被哨兵（防火墙/路由器）驱逐了。

### 现象总结
#### DNS 解析正常
`www.google.com -> 199.16.158.8` (这是 Google 的一个真实 IP，通常位于美国)。
`github.com -> 20.205.243.166` (这是 GitHub 的真实 IP，位于美国)。
VPN 成功接管了 DNS 请求，
#### 未进入VPN通道
直到第 14-17 跳，路径都显示的是 中国移动 (223.119/120.x.x) 或 教育网 的 IP。这意味着 traceroute 的数据包并没有进入 VPN 隧道！
如果走了 VPN 隧道，第 2 跳或第 3 跳之后，应该看到 VPN 服务器的虚拟 IP或者 VPN 服务商的公网 IP，而不是教育网的 115.24 和中国移动的 223.120。

### 原因分析
这是一个非常经典的“路由分离”现象。 VPN 客户端默认工作在  “代理模式 (Proxy Mode)”，而不是 “全局 TUN/TAP 模式”。

traceroute 使用的是 ICMP 协议（默认）或 UDP。而大多数 VPN 客户端（如 Clash, Surge, V2RayN）主要代理 TCP 流量（用于浏览器、下载）。对于 ICMP 流量，它们往往不代理
